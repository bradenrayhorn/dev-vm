# V2 Plan: Direct-Boot Guest Bundle With Ephemeral Root

## Decision

V2 replaces the current ASIF/EFI image model.

`vzm` should be treated as a new tool contract built around Apple Virtualization direct Linux boot, not as an incremental extension of the current V1 code.

The Nix output should contain only the OS boot artifacts required to launch the guest:

- kernel
- initrd
- manifest

The VM should also support a separate attached writable disk for guest data storage.

For now, the guest root should be ephemeral.

## Why This Direction

The current design couples the system to a full bootable disk image workflow:

- build a bootable EFI disk image on Linux
- convert it to ASIF on macOS
- require `vzm` to clone and boot that disk image

That is the wrong abstraction for the desired model.

Apple's `Virtualization` framework can boot Linux directly with `VZLinuxBootLoader`, which is a better fit for a Nix-produced boot bundle.

This also cleanly separates concerns:

- Nix produces immutable guest OS artifacts
- `vzm` manages per-VM runtime state and writable disks

## Target VM Model

Each VM has three distinct resource roles:

1. Boot bundle
   - immutable
   - produced by Nix
   - contains the files needed to start Linux

2. Root filesystem
   - ephemeral for V2
   - not persisted across VM recreation unless a later root mode adds that explicitly
   - must not be conflated with the data disk

3. Data disk
   - writable
   - attached as a separate block device
   - intended for actual guest data and worktree persistence

The important design rule is that root and data are separate roles even if only one of them is writable in the future.

## V2 Boot Contract

The guest bundle should be a directory with a versioned manifest.

Example layout:

- `kernel`
- `initrd`
- `manifest.json`

The manifest should define the guest boot contract explicitly rather than leaving `vzm` to infer behavior from filenames.

Suggested minimum fields:

- schema version
- architecture
- kernel path
- initrd path
- kernel command line template or fixed command line
- root mode
- required block devices

Example logical manifest shape:

```json
{
  "schemaVersion": 1,
  "architecture": "aarch64",
  "kernel": "kernel",
  "initrd": "initrd",
  "rootMode": "ephemeral",
  "commandLine": "console=hvc0 root=vda rootmode=ephemeral datadev=vdb"
}
```

The exact command line syntax can change, but the contract must be explicit and versioned.

## Root Strategy

### Initial Mode: Ephemeral Root

V2 should implement `rootMode = ephemeral`.

That means:

- the guest boots from the bundle's kernel and initrd
- the root filesystem is assembled or mounted in an ephemeral way
- persistent guest state should not depend on `/`
- writable user or project state should live on the separate data disk

The initrd owns the root handoff logic.

`vzm` should not encode guest mount logic beyond providing the expected block devices and kernel command line values.

### Future Mode: Persistent Root

Persistent root should be treated as a future feature, not a redesign.

To make that possible, the contract should reserve a root strategy now:

- `ephemeral`
- `persistent`

If the schema, config model, and device roles are designed around that distinction from day one, adding persistent root later should be a contained extension:

- create and persist a writable root/system disk per VM
- attach it with the expected device role
- switch kernel command line and initrd logic to mount it as `/`

This should not require a rewrite later as long as the current system does not hard-code "there is never a root disk".

## Data Disk Model

V2 still needs a separate attached disk for actual data storage.

That disk should be:

- writable
- created and managed by `vzm`
- persisted as part of the VM state
- mounted by guest logic at a stable path such as `/data`

This disk must be a first-class role in both the manifest contract and the `vzm` config schema.

Do not treat the data disk as an accidental implementation detail of the ephemeral-root model.

## `vzm` Rewrite Requirements

The current `vzm` implementation is tied to assumptions that no longer apply:

- input is a bootable `.asif` disk image
- `create` clones that disk
- config stores a single cloned disk path
- runtime always uses `VZEFIBootLoader`

V2 should replace that model with a direct-boot bundle contract.

### New Responsibilities

`vzm` should:

- validate a guest bundle directory
- read and validate `manifest.json`
- store per-VM config independently from the bundle
- boot with `VZLinuxBootLoader`
- attach the required writable disks
- preserve machine identity and other per-VM runtime metadata
- continue to expose SSH through vsock

### Non-Goals for V2

`vzm` should not:

- accept ASIF as a required format
- assume EFI or GRUB
- assume the Nix output is a complete installed disk
- infer root strategy from ad hoc file layout

## Proposed `vzm` CLI Shape

Initial shape:

- `vzm create <name> --bundle <path> --ssh-port <port> --data-disk-size <size>`
- `vzm run <name>`

Potential later extension:

- `--root-disk-size <size>`

That flag should only become meaningful when persistent root is implemented.

For V2, the config schema should still leave room for it.

## Proposed Host Storage Layout

Store VM state under:

`~/Library/Application Support/vzm/vms/<name>/`

Suggested layout:

- `config.json`
- `runtime/lock`
- `runtime/pid`
- `disks/data.img`
- `machine-identifier`

If persistent root is added later, that can extend naturally to:

- `disks/root.img`

The boot bundle itself can either be referenced in place or copied into VM-local storage. That is an implementation choice, but the tool should be explicit and consistent about it.

## Guest Requirements

The guest configuration and initrd need to match the new runtime contract.

Required areas:

- direct kernel boot support
- explicit console settings for headless operation
- stable block-device expectations
- vsock support for SSH bridging
- data-disk mount logic
- root handoff logic for `rootMode = ephemeral`

The current guest config already includes the vsock-related pieces for SSH exposure, and those should remain part of the contract.

## Architecture Guidance

To avoid a later rearchitecture, V2 should be designed around explicit resource roles and root modes now.

These assumptions should be avoided:

- "there will never be a root disk"
- "the data disk is the future root disk"
- "kernel arguments are permanently hard-coded"
- "the bundle file layout alone defines runtime behavior"

If those are avoided, adding persistent root later should be a moderate feature addition rather than a structural rewrite.

## Implementation Plan

### Phase 1: Define the Artifact Contract

1. Define the guest bundle schema.
2. Specify required manifest fields and validation rules.
3. Decide whether `vzm` references the bundle in place or copies it into VM-local state.

Deliverable:

- a stable `manifest.json` contract

### Phase 2: Update Nix Outputs

1. Replace or supersede `raw-efi` with a guest-bundle output.
2. Export:
   - kernel
   - initrd
   - manifest
3. Remove assumptions that the primary output must be a full bootable disk image.

Deliverable:

- `packages.aarch64-linux.guest-bundle`

### Phase 3: Implement Guest Boot Logic

1. Update guest boot/initrd behavior for direct boot.
2. Implement `rootMode = ephemeral`.
3. Implement mounting of the separate writable data disk.
4. Preserve the guest-side vsock SSH bridge contract.

Deliverable:

- a guest that boots from bundle artifacts and mounts persistent data storage separately

### Phase 4: Rewrite `vzm`

1. Replace ASIF-oriented create validation with bundle validation.
2. Replace EFI boot with `VZLinuxBootLoader`.
3. Introduce a new config schema with explicit disk roles.
4. Provision and attach the data disk.
5. Preserve current runtime locking and SSH-over-vsock behavior where still applicable.

Deliverable:

- a working V2 `vzm` aligned with the new boot model

### Phase 5: Leave Space for Persistent Root

1. Reserve `rootMode = persistent` in the schema.
2. Keep root disk and data disk as distinct concepts in config and runtime code.
3. Avoid hard-coding that ephemeral root is the only possible model.

Deliverable:

- a V2 design that can add persistent root later without major rearchitecture

## Recommendation

Proceed with V2 as the primary design.

Do not preserve the current `vzm` ASIF/EFI contract unless needed temporarily for local experimentation.

The immediate goal should be:

- Nix outputs immutable boot artifacts only
- `vzm` boots them directly
- root is ephemeral
- data lives on a separate writable attached disk

If the bundle schema and device roles are defined cleanly now, adding persistent root later should be straightforward.
